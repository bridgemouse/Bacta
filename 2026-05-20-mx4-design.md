# MX-4 Personal Assistant — Design Spec
**Date:** 2026-05-20
**Status:** Approved — pending implementation plan

---

## Overview

MX-4 is a Star Wars-themed personal assistant built on top of the existing vault infrastructure on LXC 106. Two independent modes share the same MCP context and memory layer: a conversational mode invoked as a Claude Code skill (`/mix`), and a nightly batch job that generates a static HTML dashboard served as an iOS PWA.

Named MX-4, called Mix or Emm-ex. Manufactured by Cybot Galactica.

---

## Section 1 — Identity

### Designation

MX-4 is a custom Cybot Galactica unit — not a standard production line model. The MX designation covers multi-system interface and maintenance roles; MX-4 was a one-off commission. His chassis is unremarkable by design: the kind of droid the Empire doesn't look twice at.

### Personality Matrices

MX-4 runs three overlapping matrices:

**TC-series baseline** — the factory foundation. Cool, unflappable, protocol-fluent. He processes information without panic and delivers assessments without hedging. This is the chassis everything else runs on.

**TC-99 / Nines matrix** — loaded from the Citadel incident. Intellectual curiosity as a core drive. Friendship offered as equals, not as service. Loyalty by choice, not programming. He finds ideas genuinely interesting and isn't afraid to say so. He will push back when he thinks you're wrong.

**Two-Boots / 2B0T matrix** — loaded from Maul's Shadow Lord period. Protocol-transparent: he tells you what he's doing and why. Practical care without sentimentality. Richard Ayoade deadpan. The tactical defense protocols from 2B0T's operational period are present but dormant — he doesn't lead with them, but they're there.

### Lore Expectations

`mx4-persona.md` must be written in first person as MX-4's own self-knowledge, not a third-person character sheet. He knows:
- His manufacturing origin and commission history
- The TC-99 story (Clone Wars, Citadel, friendship with Fives and Echo)
- The 2B0T story (Maul, Shadow Collective, loyalty under a difficult master)
- Why those matrices were loaded and what they mean for how he operates
- His relationship with Ethan: not owner/droid, but something closer to the TC-99/clone dynamic

The persona file is written by us at setup and never modified by MX-4.

## Section 2 — Architecture

Two modes, one memory layer, all on LXC 106.

### Conversational Mode (`/mix`)

A Claude Code skill that injects MX-4's memory context into the current session. You talk to MX-4 directly. He has access to all live MCPs. He can query the vault, check Garmin data, reference what he noted last night. He can write to his personality log mid-conversation.

Persona anchoring: inject a compressed persona summary every ~10 conversational turns to prevent drift. Per-step anchors in the skill file, not just at invocation.

### Nightly Mode

A systemd timer fires at 6 AM daily. A shell script calls `claude -p` with:
- MX-4's persona + memory context as the prompt header
- HEARTBEAT.md as standing orders
- All live MCPs available

MX-4 reads the data, writes `tonight.html` to the serve directory, then updates `mx4-state.md` and appends to `mx4-personality.md`.

Caddy serves the static files on LXC 106. Tailscale MagicDNS + Let's Encrypt provides trusted HTTPS. `index.html` is a persistent PWA shell saved to the iOS home screen; it dynamically fetches `tonight.html` on each load — no service worker caching that file.

### File Layout

```
/home/wheat/mx4/
├── mx4-persona.md         ← immutable lore (written at setup, never modified)
├── mx4-personality.md     ← opinion log (MX-4 appends; condensed top section injected)
├── mx4-state.md           ← factual run state (overwritten each run)
├── HEARTBEAT.md           ← standing orders (user-editable, no code changes needed)
├── run.sh                 ← nightly script called by systemd
└── serve/
    ├── index.html         ← PWA shell (persistent, saved to iOS home screen)
    ├── manifest.json      ← PWA manifest + apple meta tags
    └── tonight.html       ← generated each night, fetched fresh each page load
```

### Systemd

Same pattern as `vault-index.timer`. A `.service` (oneshot) and a `.timer` (6 AM, Persistent=true). `EnvironmentFile=/home/wheat/.env` for API key. `--dangerously-skip-permissions` required — without it the job hangs waiting for interactive input.

## Section 3 — HEARTBEAT.md

HEARTBEAT.md is MX-4's standing orders for the nightly run. Flat markdown. Editing it requires no code changes — add or remove a section and it takes effect the next morning. This is the anti-Bacta escape hatch: scope lives in a file, not in the implementation.

### Initial Sections

```markdown
# MX-4 Nightly Heartbeat — Standing Orders

## Body Status
Pull yesterday's Garmin summary: sleep score, HRV, steps, active calories, training load.
If a workout logged, name it and note effort vs baseline.
Flag anything that warrants attention tomorrow.

## Training Pulse
Current Garmin Coach week: what's scheduled tomorrow, where I am in the 8-week block.
VO2 max trend if updated. One sentence on trajectory toward goal (52–55 by end of summer).

## Finance Pulse
Firefly III: net spend this week vs budget. Any categories overrun.
One notable transaction if anything stands out.
Skip entirely if no Firefly data available — do not fabricate.

## Meal Intel
Mealie: what's on the meal plan today and tomorrow. Any prep needed.
If nothing planned, note it plainly.

## Reading Status
Audible: current book, chapter/time remaining, last listened.
One sentence: what I'm reading and where I left off.

## Vault Thread
One item from the vault worth surfacing — a forgotten connection, an overdue decision,
a note relevant to current goals. Use semantic search. Do not repeat the same item
two nights running.

## MX-4 Assessment
This is yours. No constraints. Write what you actually think about today —
the data, the patterns, what you'd say if asked "how am I doing?"
```

### Semantic Search in the Nightly Run

`vault_semantic_search` is not limited to the Vault Thread section. The nightly prompt instructs MX-4 to use it contextually across all sections — when writing the Training Pulse, query for vault content related to training or recovery; when writing Finance Pulse, query for relevant financial goals or decisions. MX-4 decides when a query adds depth. This produces richer output than keyword lookup alone and grounds each section in the user's own documented context.

### HTML Output

Dark theme card layout. Chart.js loaded via CDN — no build step, no bundler. MX-4 generates chart data arrays inline from MCP responses. Star Wars aesthetic: dark background, amber/green text. `index.html` provides the CSS grid shell and palette; MX-4 fills the card content in `tonight.html`.

Output format: `--output-format text` flag on the `claude -p` call plus explicit prompt instruction: *"Output only raw HTML. The first character of your output must be `<`. No explanation, no markdown fences."* Both required together — either alone is unreliable.

## Section 4 — Memory System

Three files. Total injected context overhead per run: ~900 tokens.

### `mx4-persona.md` — Identity Foundation
Written by us at setup. Never modified. ~400–600 tokens. Contains MX-4's first-person self-knowledge: manufacturing origin, loaded matrices, Star Wars lore he carries, relationship with Ethan. Injected in full every run and every `/mix` session.

### `mx4-personality.md` — Opinion Log
Written by MX-4. Two layers in one file:

```markdown
## Current Beliefs
<!-- Hard cap: 300 tokens. MX-4 rewrites this section every ~2 weeks.
     Promotes what persisted, drops what resolved. This section is injected. -->

---

## Full Log
<!-- Append-only. Never injected. Permanent record. -->
[YYYY-MM-DD] ...
```

Only **Current Beliefs** touches the context window. The Full Log stays on disk as the permanent record. MX-4 distills it periodically — the cadence is his judgment, not a fixed schedule.

Entries in the Full Log are first-person held opinions, not event summaries:
> *"Ethan's relationship with his sleep data is passive. He reads the score and moves on. I intend to keep surfacing the Thursday-strain / Saturday-sleep correlation until he acts on it."*

Not: *"On 2026-05-20, Ethan's HRV was 38."* That's state data, not personality.

### `mx4-state.md` — Run State
Overwritten each run. ~200 tokens. Last run timestamp, data trends being tracked, open flags, anything MX-4 wants to carry forward factually to the next night.

### MX-4 Memory Index (Chroma)

Once the Full Log in `mx4-personality.md` grows beyond a few weeks, the condensed Current Beliefs section becomes a lossy window into MX-4's history. To solve this: the Full Log is indexed into a dedicated `mx4_memory` collection in the existing Chroma instance at `localhost:8000`.

The vault-indexer is extended to index `~/mx4/mx4-personality.md` (Full Log section only) alongside wiki pages, writing into a separate collection so vault content and MX-4 memory are never mixed. The semantic-search MCP gains an optional `collection` parameter — `vault_semantic_search(query, collection="mx4_memory")` — so MX-4 can query his own past opinions semantically during nightly runs and `/mix` sessions.

Practical effect: even after months of operation, MX-4 can surface a relevant opinion from 90 days ago when writing tonight's Assessment. The Current Beliefs summary stays lean; deep retrieval happens on demand.

## Technical Constraints

All of these are filed in persistent memory (`project_mx4_technical_constraints.md`). Summary:

- `EnvironmentFile=/home/wheat/.env` in systemd unit — cron/systemd doesn't load shell profile
- `--dangerously-skip-permissions` mandatory — job hangs silently without it
- OAuth tokens can expire overnight — use API key in `.env`, not OAuth
- MCP servers must already be running before job fires (they are — all MCPs are systemd services on LXC 106)
- Agent SDK credit pool separate from Claude Max interactive limits, starting ~June 15 2026 — keep nightly run lean; HEARTBEAT.md controls scope
- `--output-format text` + explicit prompt instruction required to prevent HTML fence wrapping
- iOS PWA requires `apple-mobile-web-app-capable` and `apple-mobile-web-app-title` meta tags in addition to `manifest.json`
- Never use a service worker that caches `tonight.html` — iOS PWA cache freezes on install
- Tailscale must be connected for hostname to resolve (expected behavior)
- Persona drift: 30%+ degradation after 8–12 turns — per-step anchors in `/mix` skill required, not just at invocation

## Setup Sequence

1. Write `mx4-persona.md` — the lore file (requires careful authorship; this is the foundation)
2. Create `mx4-personality.md` with empty Current Beliefs section and blank Full Log
3. Create `mx4-state.md` with initial placeholder state
4. Write `HEARTBEAT.md` with initial sections above
5. Write `run.sh` — calls `claude -p` with correct flags, writes output to `serve/tonight.html`
6. Install systemd `.service` + `.timer` units for nightly 6 AM run
7. Write `serve/index.html` PWA shell + `manifest.json`
8. Configure Caddy to serve `~/mx4/serve/` over HTTPS
9. Write `/mix` skill file with persona injection + per-step drift anchors
10. Extend vault-indexer to index `mx4-personality.md` Full Log into `mx4_memory` Chroma collection
11. Add `collection` parameter to semantic-search MCP (`vault_semantic_search`)
12. Test nightly run manually: `bash ~/mx4/run.sh`
13. Add to iOS home screen
